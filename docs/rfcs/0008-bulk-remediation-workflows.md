# RFC 0008: Bulk Remediation Workflows

## Status

Draft

## Summary

Add backend remediation plans for uploaded lists so Reacher can partition rows into `fixed`, `safe`, `review`, and `drop` outputs without mutating the stored original data. The feature is list-centric and API-first; it produces auditable remediation artifacts and downloadable exports.

## Motivation / Problem

- List cleaning today returns verification results, but customers still have to decide how to repair or partition problematic rows.
- Common data issues such as casing, whitespace, and obvious domain typos are fixable, while other issues require review or removal.
- Teams need a repeatable remediation artifact, not just raw verification status.

## Goals

- Generate a deterministic remediation plan for a processed list.
- Separate automatic fixes from review-required rows.
- Produce exportable partitions for downstream workflows.
- Keep the original uploaded row data unchanged.

## Non-Goals

- Real-time inline editing of list rows.
- Applying transformations directly to the stored source file.
- Full support for non-list bulk jobs that do not preserve original row context.

## Current State In Reacher

- `POST /v1/lists` already preserves original row data and emits cleaned output downloads.
- Domain typo suggestions, alias normalization, and canonicalization already exist as verification-related primitives.
- There is no durable remediation plan resource or changed-row audit trail.

## Proposed Design

The remediation feature is defined for `v1` lists only.

Remediation classes are:

- `fixed`
- `safe`
- `review`
- `drop`

Automatic fixes allowed in v1 are:

- trim leading and trailing whitespace
- lowercase the domain portion
- remove zero-width and control characters
- apply the existing domain typo correction catalog

Rules that require review rather than automatic mutation are:

- catch-all
- role account
- disposable
- unknown SMTP outcome
- full inbox
- duplicate canonical email collisions

Rows are dropped automatically when they are:

- invalid syntax
- invalid recipient
- spam-trap domain
- actively suppressed

The original row is never mutated in storage. Reacher stores a remediation plan and materializes exports from that plan.

### Plan Lifecycle

A remediation plan is a durable derived artifact attached to one processed list plus one set of plan options. The lifecycle is:

1. caller requests a plan
2. Reacher resolves the effective verification result set for the list
3. deterministic remediation rules classify each row
4. plan summary and row-level actions are stored
5. callers optionally request export materialization from that plan

Plans are immutable after completion. If a caller wants a different option set or a newer rule version, Reacher creates a new plan rather than mutating the old one.

Plan identity is defined by:

- `list_id`
- `rule_version`
- normalized option set
- effective verification run digest

If those values are identical, plan creation is idempotent and returns the existing plan.

### Classification Semantics

Each row ends in exactly one class:

- `fixed`
  - safe to auto-correct and include in clean exports
- `safe`
  - no mutation needed, row is accepted as-is
- `review`
  - the row is retained, but a human should decide what to do next
- `drop`
  - the row should be excluded from clean-send outputs

The difference between `fixed` and `safe` matters for auditability. `fixed` means Reacher materially changed the outbound representation, while `safe` means it did not.

### Automatic Fix Catalog

Allowed automatic fixes are intentionally narrow in v1:

- trim leading and trailing ASCII and Unicode whitespace
- lowercase the domain
- strip zero-width and disallowed control characters
- normalize Unicode punctuation that breaks parsing
- correct domains through the existing typo catalog only when the correction has one unambiguous replacement

Explicitly not allowed in v1:

- guessing the local part
- rewriting plus tags
- merging duplicate rows into one canonical row
- editing arbitrary CSV columns beyond the email field and later optional name/domain helper fields

### Review And Drop Rules

Rows are forced to `review` when:

- the email is deliverable but operationally sensitive
- the fix would change meaning rather than formatting
- multiple plausible typo corrections exist
- duplicate canonical emails exist and `collapse_duplicates=false`

Rows are forced to `drop` when:

- the email is structurally invalid
- the domain is confirmed spam-trap
- the recipient is invalid
- the email is actively suppressed and `drop_suppressed=true`

### Output Partitions

Exports are generated by partition:

- `fixed.csv`
- `safe.csv`
- `review.csv`
- `drop.csv`
- optional `combined_clean.csv` containing `fixed + safe`

Each export row includes provenance columns appended to the original schema:

- `_reacher_classification`
- `_reacher_rule_id`
- `_reacher_confidence`
- `_reacher_original_email`
- `_reacher_effective_email`

This keeps exports operationally useful without hiding what changed.

## Public API / Interface Changes

Add remediation endpoints:

- `POST /v1/lists/{list_id}/remediation-plan`
- `GET /v1/lists/{list_id}/remediation-plan`
- `POST /v1/lists/{list_id}/remediation-exports`
- `GET /v1/lists/{list_id}/remediation-exports/{export_id}/download`

`POST /v1/lists/{list_id}/remediation-plan` request fields:

- `allow_partial`
- `apply_domain_typos`
- `drop_suppressed`
- `collapse_duplicates`

Plan responses include:

- `plan_id`
- `status`
- `summary_counts`
- `rule_version`
- `generated_at`

Each remediation row includes:

- `row_number`
- `classification`
- `before`
- `after`
- `rule_id`
- `confidence`

Additional plan response fields:

- `rule_version`
- `effective_job_id`
- `row_counts`
- `options`
- `result_state_digest`
- `created_at`
- `completed_at`

Example plan create request:

```json
{
  "allow_partial": false,
  "apply_domain_typos": true,
  "drop_suppressed": true,
  "collapse_duplicates": false
}
```

Example plan summary response:

```json
{
  "plan_id": "rmp_01HT...",
  "list_id": "lst_01HT...",
  "status": "completed",
  "rule_version": "remediation_v1",
  "effective_job_id": "job_01HT...",
  "summary_counts": {
    "fixed": 84,
    "safe": 702,
    "review": 91,
    "drop": 23
  },
  "options": {
    "allow_partial": false,
    "apply_domain_typos": true,
    "drop_suppressed": true,
    "collapse_duplicates": false
  },
  "generated_at": "2026-03-30T18:42:00Z"
}
```

`POST /v1/lists/{list_id}/remediation-exports` accepts:

- `plan_id`
- `partitions[]`
- `format` default `csv`

Only `csv` is in scope for v1. `ndjson` is deferred.

## Data Model / Storage Changes

Add tables:

- `v1_remediation_plans`
- `v1_remediation_rows`
- `v1_remediation_exports`

Storage decisions:

- `before` and `after` are stored as JSON objects keyed by original column name.
- `rule_id` identifies the deterministic remediation rule used.
- Export rows reference the plan that produced them and are immutable after creation.

`v1_remediation_plans` columns:

- `id`
- `tenant_id`
- `list_id`
- `effective_job_id`
- `rule_version`
- `options JSONB`
- `result_state_digest`
- `status`
- `summary_counts JSONB`
- `created_at`
- `completed_at`

`v1_remediation_rows` columns:

- `id`
- `tenant_id`
- `plan_id`
- `row_number`
- `classification`
- `rule_id`
- `confidence`
- `before JSONB`
- `after JSONB`
- `created_at`

`v1_remediation_exports` columns:

- `id`
- `tenant_id`
- `plan_id`
- `format`
- `partitions JSONB`
- `storage_key`
- `created_at`

Indexes:

- unique `(tenant_id, list_id, rule_version, result_state_digest, options_hash)` on plans
- btree `(tenant_id, plan_id, classification, row_number)` on remediation rows
- btree `(tenant_id, list_id, created_at desc)` on plans

Retention decision:

- plans and row details follow normal list retention unless a tenant deletes the source list
- exports may be garbage-collected earlier using the same pattern as existing download artifacts

## Auth, Permissions, And Tenant Isolation

- All remediation APIs require the existing `lists` scope.
- A plan may only be created for a list owned by the caller’s tenant.
- Export downloads use the same tenant checks as current list downloads.

Additional rules:

- plan generation and export creation are both list-write behaviors, but read-only retrieval of an existing plan still uses the current `lists` scope rather than introducing a separate remediation scope
- callers cannot create a plan against a list they can read only through a future shared-view concept; cross-tenant sharing is out of scope

## Failure Modes And Edge Cases

- Creating a plan before list processing completes returns `409` unless `allow_partial = true`.
- If a plan already exists for the same list and identical options, Reacher returns the cached plan rather than creating a duplicate.
- If the rule catalog version changes, old plans remain valid but are marked with the earlier `rule_version`.
- Duplicate canonical email collisions never auto-merge two rows; they are routed to `review`.

Additional cases:

- if the original upload had no email column mappable to verification results, plan creation returns `422`
- if partial mode is enabled, the response marks the plan as provisional and includes `processed_rows` vs `total_rows`
- if a later retry changes result quality for some rows, the old plan remains readable but a new request generates a new plan because the `result_state_digest` changed
- if storage upload for exports fails, the plan remains complete and callers may retry export creation without recomputing row classifications

## Alternatives Considered

- Mutating list rows in place. Rejected because it destroys auditability and user trust.
- Returning remediation hints only inline in list downloads. Rejected because plans and exports need stable IDs and repeatability.
- Supporting raw non-list bulk jobs first. Rejected because list uploads already preserve the row context remediation needs.

## Rollout / Migration / Compatibility

- Ship as additive list-only APIs.
- Reuse the existing list download format for export generation where possible.
- Keep current list processing behavior unchanged unless a remediation plan is requested.

Rollout plan:

1. add tables and deterministic rule engine
2. ship plan create/get endpoints
3. ship partitioned export creation and download
4. document provenance columns and operational usage

Compatibility guarantees:

- existing list upload and download behavior is unchanged until remediation routes are called
- the source list data remains immutable
- remediation artifacts are additive and can be ignored by existing clients

## Metrics And Success Criteria

- Percentage of processed lists with a remediation plan.
- Share of rows landing in `fixed`, `safe`, `review`, and `drop`.
- Automatic fix acceptance rate measured by export generation.
- Time from plan request to export readiness.

Additional success criteria:

- support requests about "which rows should I fix or remove" can be answered via plan outputs rather than manual CSV manipulation
- automatic fixes remain explainable with a bounded rule catalog rather than opaque transformation logic

## Test Strategy

- Plan generation tests on processed and partially processed lists.
- Rule-classification tests for all automatic, review, and drop classes.
- Export partition tests.
- Idempotent plan regeneration tests.
- Tenant isolation tests on list ownership and download access.

Add explicit coverage for:

- one-to-many typo candidates producing `review` instead of auto-fix
- duplicate canonical emails with and without `collapse_duplicates`
- preservation of non-email columns in exports
- export retry after transient storage failure
- provisional plan generation for partial lists

## Unresolved Risks

- Automatic domain typo correction can produce false positives if the correction catalog is too aggressive.
- Large plans can create significant storage overhead without later retention controls.

The largest product risk is overtrust in automatic fixes. That is why the initial fix catalog is deliberately narrow and every changed row includes before/after provenance.
